# Track Specification: samples/roboface2.py クラス構造の整理

## 1. 概要
`samples/roboface2.py` におけるロボットの顔制御ロジックを、`RobotFace` クラスを起点とした明確な責務分担に基づく構造にリファクタリングする。将来的なファイル分割を容易にし、名前付けの一貫性とクラス間の独立性を高める。

## 2. 機能要件・設計方針
### 2.1. RobotFace クラスの再定義（処理の起点）
- **状態管理と遷移**: 顔の状態（`RfState`）を保持し、表情間の遷移（補間）ロジックを管理する。
- **タイミング制御**: `update()` メソッドを通じて、アニメーションの進行（経過時間の管理）を自律的に行う。
- **外部注入型インターフェース**: 描画先（`PIL.Image`）や出力デバイス（`DisplayBase`）は、メソッド引数や初期化時に外部から渡す（または依存しない）設計とする。

### 2.2. 役割分担の整理
- **Model (`RfState` / `RfBrowState`, `RfEyeState`, `RfMouthState`)**: 顔の幾何学的な数値データ。
- **Controller (`RobotFace`)**: 顔の「状態」と「時間軸」の管理主体。
- **Renderer (`RfRenderer`)**: `RfState` を受け取り、`PIL.Image` に描画を実行する。
- **Device Interface (`DisplayBase`, `Lcd`, `CV2Disp`)**: 生成された画像を物理画面に出力する。

### 2.3. コード品質と拡張性
- **命名規則の統一**: `Rf` プレフィックスによる一貫したネーミング。
- **エントリポイントの分離**: コアロジックと CLI 処理（`click`）の境界を明確にする。
- **設定の集約**: `RfConfig` クラス等に定数を集約する。

## 3. 非機能要件
- **ファイル分割の準備**: クラス間の疎結合化。
- **既存機能の維持**: 現行の `RandomMode` および `InteractiveMode` の動作維持。

## 4. 受入基準
- [ ] `RobotFace` が状態更新とアニメーションの起点として機能している。
- [ ] クラス名、メソッド名、定数名が `Rf` を含む一貫したルールで命名されている。
- [ ] `mise run lint` および `mise run test` がパスする。
- [ ] 既存のサンプル動作（ランダム表示、対話モード）が正常に機能する。
